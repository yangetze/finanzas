## On the Horizon

### Full Sprint Autonomy With TDD
Hand an entire sprint backlog to a single agent that reads tickets, writes failing tests, implements until green, and opens PRs for each feature in sequence. The agent self-verifies against the test suite before ever requesting review, turning a multi-day sprint into a supervised overnight run. You wake up to a stack of green, mergeable PRs.

**Getting started:** Use the Bash and Write tools to drive a red-green-refactor loop and the GitHub MCP server (`create_pull_request`) to ship each feature as its own PR.

> _"Here is our sprint backlog (tickets below). For each ticket, work strictly test-first: write failing tests, implement until the full suite passes, refactor, then open a dedicated pull request with a clear description linking the ticket. Run the test suite after every change and do not open a PR until tests are green. Process tickets sequentially and give me a final summary table of PR links and test results."_

### Event-Driven CI Instead of Polling
Replace wasteful scheduled wakeups that check CI after merge with a reactive workflow where the agent subscribes to CI status and only acts on real state changes. The agent watches a PR's checks, reacts the moment they pass or fail, auto-fixes failures, and merges—never burning cycles polling stale state.

**Getting started:** Use `mcp__github__pull_request_read` to poll check status on-demand within a single loop and `merge_pull_request` only once checks report success, avoiding standalone scheduled wakeups.

> _"For each open PR, monitor its CI checks by reading PR status in a tight loop within this session—do not schedule separate future wakeups. When checks fail, diagnose the logs, push a fix, and re-check. Only call merge_pull_request once all required checks pass. If a PR is already merged, skip it immediately. Report which PRs merged and which needed fixes."_

### Parallel Agents Across Feature Branches
Spin up multiple agents working simultaneously on independent features, each on its own branch with its own test loop, then converge their PRs once CI is green. Parallelization collapses a sequential sprint into concurrent streams, with a coordinator agent resolving merge order and conflicts.

**Getting started:** Decompose the backlog into independent units, dispatch each to a subagent with isolated git worktrees via Bash, and use the GitHub MCP tools to manage and merge their PRs in dependency order.

> _"Analyze this backlog and identify which tickets are independent and can be built in parallel versus those with dependencies. For each independent ticket, create a separate git worktree and branch, implement it test-first in isolation, and open a PR. Then determine the correct merge order based on dependencies, resolve any conflicts, and merge them sequentially once CI passes. Show me the dependency graph you used."_
